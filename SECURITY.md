# DrewRPG Level System Security Policy

DrewRPG Level System prioritizes data integrity and the stability of your server. We take all potential security vulnerabilities within the plugin seriously and actively encourage Responsible Disclosure.

## 1. Security Vulnerability Reporting (How to Report a Vulnerability)
If you discover a security vulnerability in the plugin code, configuration file handling, or command processing, please do not disclose the issue publicly. You must report it via the private channel below before sharing it in server operator communities or on public platforms like GitHub Issues.

### Private Reporting Channel
* Email: drew0414drew@gmail.com
  * Please include **"[SECURITY REPORT]"** in the subject line of your email.

### Required Information for Reporting
* **Vulnerability Summary:** A brief description of the type of vulnerability (e.g., privilege escalation, potential data leak, command injection, etc.).
* **Reproduction Steps:** Detailed steps necessary to reproduce the issue (including command syntax, specific JSON settings, etc.).
* **Impact:** An explanation of the potential risks if the vulnerability were to be exploited.
* **Environment:** Specify the server software (Paper, Spigot, etc.) version and the plugin version (V 1.0.0) you are using.

## 2. Response and Remediation Process
We strive to process all security reports as quickly as possible.
* **Report Acknowledgment:** We aim to acknowledge receipt of the report within 48 hours.
* **Vulnerability Assessment:** The vulnerability will be rigorously tested, with the timeline depending on the complexity of the vulnerability.
* **Patch Release:** The new version (e.g., V 1.0.1) containing the patch will be released immediately upon completion via GitHub Releases.

## 3. Supported Versions
The following version is currently supported for receiving security patches and updates:
* V 1.0.0: Currently Supported

# DrewRPG Level System 보안 정책 (Security Policy)

DrewRPG Level System은 데이터 무결성과 사용자 서버의 안정성을 최우선으로 생각합니다. 저희는 플러그인에서 발생할 수 있는 잠재적인 보안 취약점을 매우 중요하게 다루고 있으며, 책임 있는 보안 취약점 공개(Responsible Disclosure)를 적극 권장합니다.

## 1. 보안 취약점 보고 방법 (How to Report a Vulnerability)
플러그인 코드, 설정 파일 처리, 명령어 처리 등에서 보안 취약점을 발견하신 경우, 절대 공개적으로 문제를 공유하지 말아 주십시오. 서버 운영자 커뮤니티나 GitHub Issues에 공개하기 전에 반드시 아래의 비공개 채널을 통해 보고해 주셔야 합니다.

### 비공개 보고 채널
* 이메일: drew0414drew@gmail.com
  * 보고 시 메일 제목에 **"[SECURITY REPORT]"**를 명시해 주십시오.

### 보고 시 포함해야 할 정보
* **취약점 요약:** 어떤 종류의 취약점인지 간략하게 설명합니다 (예: 권한 상승, 데이터 유출 가능성, 명령어 인젝션 등).
* **재현 단계:** 문제를 재현할 수 있는 상세한 단계 (명령어 구문, 특정 JSON 파일 설정 등)를 포함합니다.
* **영향:** 취약점을 악용했을 때 발생할 수 있는 잠재적 위험을 설명합니다.
* **환경:** 사용 중인 서버 소프트웨어(Paper, Spigot 등) 버전과 플러그인 버전(V 1.0.0)을 명시해 주십시오.

## 2. 대응 및 조치 절차 (Response and Remediation Process)
저희는 모든 보안 보고를 가능한 한 신속하게 처리하기 위해 노력합니다.
* **보고 접수 확인:** 보고서 접수 후 48시간 이내에 접수 확인을 목표로 합니다.
* **취약점 평가:** 취약점의 복잡성에 따라 타임라인이 달라질 수 있지만, 철저하게 테스트를 진행합니다.
* **패치 릴리스:** 패치를 포함한 새 버전(예: V 1.0.1)은 완료 즉시 GitHub Releases를 통해 배포됩니다.
